> **更新**: 2026-03-22
> **适用范围**: `plan.md`、`docs/dev_plan.md`、`docs/frontend_dev_plan.md` 及项目实现决策

## 1. 文档定位

### 1.1 这份文档解决什么问题

本文件用于沉淀 **长期稳定、跨迭代复用** 的项目原则，避免把原则散落在：

- `plan.md` 的里程碑说明
- `docs/dev_plan.md` 的当前轮优先级
- `docs/frontend_dev_plan.md` 的前端执行细则

它回答的问题是：

- 这个项目“什么算对的方向”
- 当工程效率、历史真实性、可玩性、视觉表现冲突时，优先级如何排序
- 团队在做新功能、重构和 UI 迭代时，应该用什么标准判断取舍

### 1.2 与其他文档的关系

- `plan.md`：产品里程碑与阶段交付物
- `docs/dev_plan.md`：当前技术优先级与扫描结果
- `docs/frontend_dev_plan.md`：当前前端优先级与界面落地路线
- `docs/development_principles.md`：长期稳定的项目级原则，不承载短期任务拆解

### 1.3 应写什么，不应写什么

适合写在这里的内容：

- 长期有效的设计哲学
- 工程边界与架构约束
- 玩法、可读性、反馈、节奏等项目级判断标准

不适合写在这里的内容：

- 本周 / 本轮待办事项
- 某一个功能的详细实现步骤
- 会随迭代频繁变化的优先级排序

## 2. 核心产品原则

### 2.1 核心循环优先（Core Loop First）

**为什么**

《Cent Jours》的价值不在于“系统很多”，而在于“每天一回合”的决策循环本身有张力。

**判断标准**

- 玩家是否能在 30 秒到 2 分钟内完成一轮“看情报 → 做选择 → 承担后果”
- 即使没有最终美术资源，这一轮是否仍然成立

**对本项目的直接含义**

- 新功能优先服务 Dawn / Action / Dusk 循环
- 不能为了外围系统拖慢主循环验证

### 2.2 有趣选择与可见后果

**为什么**

策略游戏的乐趣来自取舍，而不是来自隐藏信息后的盲猜。

**判断标准**

- 一个选择是否同时包含收益、代价和上下文信息
- 玩家是否能理解“为什么结果是这样”

**对本项目的直接含义**

- 政策、战斗、忠诚度、历史事件必须能被解释
- UI 必须帮助玩家理解选择后的变化

### 2.3 玩家心理优先于绝对写实

**为什么**

历史真实感重要，但策略游戏首先必须让玩家感觉公平、清楚、可掌控。

**判断标准**

- 玩家失败后能否归因
- 玩家是否会觉得系统“乱”“黑箱”“被坑”

**对本项目的直接含义**

- 命令偏差、随机性、冷却、事件触发都要可读
- 不为了写实牺牲可理解性和公平感

### 2.4 紧迫感优先于复杂度堆砌

**为什么**

本项目的差异化核心之一是“100 天、每天都重要”的压缩张力。

**判断标准**

- 新系统是否强化了“有限时间内做艰难决策”
- 是否只是增加了操作量和阅读量

**对本项目的直接含义**

- 不引入削弱节奏的低价值复杂度
- 任何新系统都要服务“时间压力”

### 2.5 历史感服务于游戏体验

**为什么**

历史题材是魅力来源，但不应压倒玩家体验本身。

**判断标准**

- 历史事件、人物、文本是否强化了游戏情绪与策略张力
- 是否只是堆资料感而没有玩法价值

**对本项目的直接含义**

- 叙事、日记、历史节点必须和机制相互支撑
- 历史元素优先做“可感知价值”，而不是“百科全书”

## 3. 游戏设计原则

### 3.1 目标—障碍—工具

**为什么**

游戏的本质不是给答案，而是设计值得解决的问题。

**判断标准**

- 玩家目标是否明确
- 障碍是否真实存在
- 工具是否足够但不过度

**对本项目的直接含义**

- 每个阶段都要明确告诉玩家：现在该做什么、为什么难、手里有什么手段

### 3.2 可读性与可预判性

**为什么**

复杂系统不等于复杂表达。玩家必须能快速读懂状态与趋势。

**判断标准**

- 重要信息是否一眼可见
- 下一步风险是否能提前预判

**对本项目的直接含义**

- 顶栏、边栏、地图、托盘必须形成稳定的信息层级
- 关键状态变化需要明确反馈

### 3.3 教学嵌入游玩（Teach Through Play）

**为什么**

复杂策略游戏也不应把理解成本全部推给说明文本。

**判断标准**

- 玩家是否能通过前几回合自然学会系统
- 是否需要长篇阅读才能开始玩

**对本项目的直接含义**

- 首批可玩路径必须足够短
- UI 和反馈要承担教学责任

### 3.4 反馈至上（Game Feel / Juice）

**为什么**

每次点击、选择、结算都应该有存在感，否则作品会像“能跑的面板”。

**判断标准**

- 玩家操作后是否立刻收到视觉、文字、音效或状态反馈
- 变化是否足够明显、足够及时

**对本项目的直接含义**

- 选中态、数值闪烁、结算结果、游戏结束画面都属于核心体验，不是装饰

### 3.5 失败必须可归因

**为什么**

“输了但知道为什么”会驱动再来一局；“输了但不懂为什么”只会流失。

**判断标准**

- 玩家能否复盘关键失败原因
- 系统是否提供了足够证据

**对本项目的直接含义**

- 政治崩溃、战败、叛逃、结局都要有清晰解释

## 4. 前端与体验原则

### 4.1 原型先行

**为什么**

先验证结构和节奏，再投入大量视觉资源。

**判断标准**

- 没有最终美术时，界面结构是否仍成立

**对本项目的直接含义**

- 先做可展示主场景，再做高保真美术

### 4.2 先骨架，后润色

**为什么**

布局、交互和数据流没站稳之前，细节优化收益很低。

**判断标准**

- 当前问题是“看不懂 / 不能用”，还是“还不够精致”

**对本项目的直接含义**

- 优先修结构问题，而不是先磨像素细节

### 4.3 Godot 薄展示层

**为什么**

前端应该负责展示与交互，而不是成为第二个游戏引擎。

**判断标准**

- 无 UI 时仍然有意义的逻辑，不应写在 GDScript 场景脚本里

**对本项目的直接含义**

- 规则在 Rust，引擎状态通过 `TurnManager` → `GameState` → UI 流动

### 4.4 组件复用优先

**为什么**

同类交互和视觉重复造轮子会让 UI 失控。

**判断标准**

- 现有组件是否已能覆盖 70% 以上需求

**对本项目的直接含义**

- 优先复用 `rn_slider.gd`、`decision_card.gd`、主题系统

### 4.5 信息层级优先

**为什么**

策略游戏 UI 的首要任务是帮助玩家判断，而不是填满屏幕。

**判断标准**

- 最重要的信息是否位于最显著的位置
- 次级信息是否不会干扰主判断

**对本项目的直接含义**

- 顶栏负责全局状态，边栏负责解释与摘要，托盘负责选择

### 4.6 结构问题优先于像素补丁

**为什么**

UI 裁切、溢出和挤压，很多时候不是“再调 8 个像素”就能解决，而是容器分配、最小高度、滚动策略或组件内部约束出了问题。误判成像素问题，会直接浪费迭代轮次。

**判断标准**

- 当前问题是由布局模型导致，还是由局部边距/字号导致
- 是否已经检查过 `size_flags`、`custom_minimum_size`、滚动容器和组件内部最小尺寸
- 同一问题是否已经连续两轮仍未收口

**对本项目的直接含义**

- 顶栏裁切、底部贴边、Sidebar 挤压这类问题，先检查布局模型，再调像素值
- 若连续两轮局部补丁无效，应立即升级为“最小高度 + 弹性分配 + 局部滚动”的结构性重构
- 前端组件在响应父级尺寸时，应同步检查内部 UI 是否会真正重建

## 5. 工程与架构原则

### 5.1 TDD

**为什么**

策略游戏的规则链路长、状态耦合强。没有测试保护时，平衡调整和小修复都很容易带出隐藏回归。

**判断标准**

- 规则或状态变更是否先有测试或明确验收检查项
- 重构后是否仍能由自动化验证快速确认行为未变

**对本项目的直接含义**

- Rust 规则层默认遵循 `Red -> Green -> Refactor`
- 若本轮无法先补测试，至少要在计划文档中写清最小验收检查项
- `cargo test` 通过是提交业务逻辑前的默认门槛

### 5.2 单一状态源

**为什么**

策略游戏最怕“两个都像是真的状态”。一旦多个层都在维护权威数值，Bug 会先表现成 UI 错乱，再演变成规则失真。

**判断标准**

- 同一运行时数值是否只有一个权威来源
- 前端是否只缓存和呈现，而不平行计算核心结果

**对本项目的直接含义**

- Rust 引擎是真实规则状态
- `TurnManager` 负责同步，`GameState` 是 UI 只读缓存
- Godot 侧不自行推导新的合法性、冷却或战斗结果

### 5.3 GDScript 薄层

**为什么**

Godot 擅长场景组织、交互与展示，但不适合作为第二套核心业务引擎。把规则留在 Rust，才能保证边界清晰和长期可维护。

**判断标准**

- 一段逻辑在没有 Godot UI 时是否仍然“有意义”
- 如果有意义，它是否被放在 Rust 或明确的状态层，而不是场景脚本

**对本项目的直接含义**

- GDScript 负责 UI、信号、桥接和展示元数据
- 核心规则、事件效果、战斗计算、冷却结算应留在 Rust
- Godot 层可以组合数据，但不复制规则引擎

### 5.4 DRY

**为什么**

重复定义会最快地制造漂移。游戏项目里，规则、键名、元数据和数值只要多处维护，之后就一定会有一处忘改。

**判断标准**

- 修改一条规则是否需要同时改多个实现文件
- 相同结构或键名是否只在一个权威位置定义

**对本项目的直接含义**

- 战斗、政治、事件计算只在一处权威实现
- 前端可保留展示文案，但不复制核心公式
- GDExtension 返回键名和状态结构要避免多处散写

### 5.5 常量集中管理，避免硬编码

**为什么**

散落的魔法数字、键名和阈值最容易制造漂移。它们短期写起来快，但一旦跨文件、跨层同时存在，后续就会变成“改了一处、漏了三处”。

**判断标准**

- 同一尺寸、阈值、键名、显示上限是否在多个文件重复出现
- 该常量是否可以从主题 token、数据表、配置或单一导出模块取得
- 当前写死的值是否只是临时补丁，而不是稳定规则

**对本项目的直接含义**

- 游戏规则常量优先在 Rust 权威层集中定义，再通过接口暴露给 Godot
- 前端尺寸、颜色、文案 key、显示上限应优先从主题、布局规则或单一配置推导
- 冷却时间、忠诚度阈值、阶段键名、状态字段名不允许在多个脚本散写
- 若某个硬编码值无法立即抽离，至少要限制在单一文件单一位置，并标注后续收敛方向

### 5.6 YAGNI

**为什么**

游戏开发很容易为“也许以后会用到”提前抽象，结果把时间花在不存在的问题上，而不是把核心循环做扎实。

**判断标准**

- 当前改动是否解决了一个已经存在的问题
- 抽象是否由现实重复逼出来，而不是由未来想象驱动

**对本项目的直接含义**

- 在主场景、核心循环、状态同步还未稳定前，不提前搭过大的前端框架
- 三行可读重复通常好过一个过早的万能系统
- 优先支撑当前演示和验证，而不是未来理想形态

### 5.7 KISS

**为什么**

复杂策略体验不等于复杂实现。实现层越绕，越难定位问题，也越难让后续迭代理解并接手。

**判断标准**

- 是否存在更短、更清晰、更容易维护的实现路径
- 为了“优雅”引入的间接层是否真的提升了可读性

**对本项目的直接含义**

- 数据流保持单向：引擎 -> `TurnManager` / `GameState` -> UI
- 场景脚本优先清晰直白，少做炫技式抽象
- 小量重复优于难读的通用化包装

### 5.8 数据驱动设计

**为什么**

策略游戏的角色、事件、地图、平衡参数会持续变化。把这些内容硬编码进逻辑层，会让每次调整都变成高成本改代码。

**判断标准**

- 角色、事件、地图、视觉 token 等是否能在外部数据中调整
- 策划和平衡修改是否尽量不需要改引擎代码

**对本项目的直接含义**

- `characters.json`、`historical.json`、`map_nodes.json`、`design_tokens.json` 应持续作为核心数据源
- 前端元数据如可稳定外提，应优先外提而不是散落脚本
- 数据结构的变化要优先体现在契约，而不是隐式埋在 UI 逻辑里

### 5.9 边界契约注释

**为什么**

Rust 与 GDScript 之间大量通过 `Dictionary` / `Array` 传值，缺少编译期字段保护。没有边界说明时，新开发者很难判断返回结构和可用键名。

**判断标准**

- 调用侧是否能快速看懂返回值结构
- 新同事是否不用反复跳到 Rust 源码也能安全接 UI

**对本项目的直接含义**

- GDExtension 暴露给 GDScript 的结构应有轻量契约注释
- 返回键名、类型和关键语义要在调用侧或桥接层写清
- 边界注释不是形式主义，而是跨语言维护成本控制

### 5.10 ADR 记录重大决策

**为什么**

真正耗团队时间的，不是写文档，而是反复重新争论已经做过的架构决策。ADR 的作用是把“为什么这么做”稳定下来。

**判断标准**

- 是否属于跨迭代、会影响后续实现方式的重要选择
- 若未来回看，是否需要知道当时为何拒绝了其他方案

**对本项目的直接含义**

- Rust + GDExtension、只读缓存、政策冷却接口等决策应通过 ADR 固化
- 新的跨层架构调整、状态流改造、重要 UI 接口变化也应继续记录
- ADR 用来减少反复讨论，不是增加流程负担

## 6. 迭代与验证原则

### 6.1 原型先于资产投入

**为什么**

游戏开发里，最容易浪费的是在玩法未站稳前就投入大量美术、动画和包装。先验证“好不好玩”，再投入高成本资源，能显著降低返工。

**判断标准**

- 在没有最终美术资产时，核心体验是否仍然成立
- 当前投入是否在验证玩法，而不是在掩盖玩法问题

**对本项目的直接含义**

- 新机制先用占位 UI、开发态文案和假数据跑通
- 在主循环、地图交互、战斗反馈未稳定前，不优先投入高成本美术包装

### 6.2 垂直切片优先

**为什么**

单独把每个系统都做一点，很容易看起来“都在推进”，但真正的集成问题会一直被推迟。垂直切片能尽早暴露接口、节奏和体验缺口。

**判断标准**

- 当前工作是否形成了一条从输入到反馈的完整链路
- 是否能把本轮成果真正跑起来，而不是只多了几个孤立模块

**对本项目的直接含义**

- 优先做“可跑的一整条链路”，例如政策选择 -> 引擎结算 -> UI 刷新 -> 叙事反馈
- 地图、战斗、忠诚度等新系统应优先接成一条可演示闭环，而不是先各做半截

### 6.3 小步迭代，尽快试玩

**为什么**

“好玩”是验证出来的，不是推理出来的。迭代步子过大时，问题来源会变得难以定位，试玩反馈也会失焦。

**判断标准**

- 单轮改动是否足够小，能明确归因好坏
- 每轮完成后是否能快速试玩或复核，而不是积压多轮再统一检查

**对本项目的直接含义**

- 优先拆成可单独提交、可单独验收的小轮次
- 每一轮都应能快速验证：变得更可玩了，还是只是更复杂了

### 6.4 测试验证与试玩验证并重

**为什么**

自动化测试能证明“行为正确”，但不能完整证明“体验成立”。游戏开发必须同时关心正确性和感受。

**判断标准**

- 可自动化的部分是否已尽量自动化
- 交互、反馈、节奏、可读性是否通过试玩验证过

**对本项目的直接含义**

- Rust 规则层优先用测试覆盖输入/输出契约
- Godot 主场景、交互反馈、信息可读性需要通过实际运行和试玩确认
- 不是所有内容都要手动测，但涉及体验质量的部分不能只靠测试代替

### 6.5 每轮必须有可见成果

**为什么**

可见成果既是开发节奏控制，也是协作成本控制。没有可见结果的迭代，很容易让计划文档和实际进展脱节。

**判断标准**

- 本轮结束后，是否能明确展示“新增了什么”
- 旁观者是否能通过界面、日志、测试结果或文档变化看出增量

**对本项目的直接含义**

- 每轮至少产出一种可观察结果：场景变化、交互闭环、测试新增、文档状态更新
- 如果一轮结束后既不能运行、也不能演示、也没有验证结果，就说明任务切得还不够好

### 6.6 视觉问题以目标平台真机收口

**为什么**

无头测试能发现解析和加载错误，但不能证明布局、裁切、滚动和信息层级已经对了。UI 问题如果不尽早回到目标平台截图验证，很容易在命令行通过、在真机上继续出错。

**判断标准**

- 当前问题是否属于视觉布局、裁切、滚动、文本可读性或交互反馈
- 是否已经拿到目标平台、目标分辨率下的运行截图或手动验收结果
- 是否错误地把无头测试当成了视觉验收替代品

**对本项目的直接含义**

- Windows Godot `1280x720` 运行截图是当前前端布局问题的默认收口标准
- `godot --headless` 只用于脚本解析、资源装载和扩展初始化兜底
- 若目标截图与命令行结论冲突，以目标截图暴露的问题为准

> 本章不等于“必须全部手动测试”。更准确地说：自动化验证优先覆盖可自动化部分，开发者手动试玩用于验证可玩性、可读性与反馈质量。

## 9. 执行机制

本章定义：如何确保这些原则在实际开发中被真正应用，而不是只停留在文档里。

### 9.1 第一层：原则分级

原则不应被同等对待，而应分成三类：

- **A 类：硬约束原则**
  - 每次开发都必须检查
  - 违反后默认视为设计或架构问题，而不是“可接受偏差”
  - 当前默认包括：
    - GDScript 薄层
    - 单一状态源
    - 数据驱动设计
    - 文档同步

- **B 类：默认适用原则**
  - 大多数任务都应检查
  - 只有在明确不相关时才可跳过
  - 当前默认包括：
    - 核心循环优先
    - 有趣选择与可见后果
    - 可读性与可预判性
    - 先骨架，后润色

- **C 类：按任务触发原则**
  - 只在特定任务中重点检查
  - 当前默认包括：
    - 反馈至上
    - 教学嵌入游玩
    - 垂直切片优先
    - 历史感服务于游戏体验

### 9.2 第二层：开工前动态选择适用原则

每个任务开工前，**不预设固定数量**，而是由 AI 根据本轮改动内容动态选择“本轮适用原则”。

规则如下：

- 至少选择 1 条
- 通常为 2-5 条
- 若任务涉及跨层、跨系统或玩法改动，可超过 5 条
- 不允许机械地每轮固定选择 3 条

动态选择逻辑示例：

- **Rust 规则 / 数值 / 事件改动**
  - 核心循环优先
  - 有趣选择与可见后果
  - 数据驱动设计
  - TDD

- **Godot UI / 交互改动**
  - 可读性与可预判性
  - 反馈至上
  - 信息层级优先
  - Godot 薄展示层

- **新手体验 / 前几回合节奏改动**
  - 教学嵌入游玩
  - 玩家心理优先于绝对写实
  - 核心循环优先

- **跨层接口 / 状态同步改动**
  - 单一状态源
  - 边界契约注释
  - GDScript 薄层
  - DRY

输出要求：

- 在开始实现前，明确写出“本轮适用原则”
- 若某条常用原则本轮明确不适用，也应允许跳过，不强行凑数

### 9.3 第三层：提交前固定检查维度

提交前不固定“5 个具体问题”，而是固定 **5 个检查维度**。  
AI 应根据本轮任务复杂度，围绕这 5 个维度动态生成检查问题。

固定维度如下：

1. **产品 / 玩法价值**
   - 这次改动是否强化了核心循环、选择质量或节奏张力

2. **玩家体验 / 可读性**
   - 玩家是否能看懂发生了什么、为什么会这样、下一步该做什么

3. **架构边界**
   - 是否违反薄层、单一状态源、数据驱动或分层职责

4. **验证闭环**
   - 这次改动是否有可测试、可试玩、可观察的验证方式

5. **文档同步**
   - 是否需要同步更新 `plan.md`、`docs/dev_plan.md`、`docs/frontend_dev_plan.md`

说明：

- 固定的是维度，不是问题数量
- 简单任务可能每个维度只需 1 个问题
- 复杂任务可以围绕某个维度展开多个问题

### 9.4 第四层：能自动化的尽量自动化

原则不应只靠人工记忆。凡是可自动化检查的，优先自动化。

当前项目建议自动化的内容：

- **Rust 改动**
  - 自动运行 `cargo test`
  - 必要时补充目标模块测试

- **GDScript / 场景改动**
  - 至少做一次 Godot 项目加载或场景解析检查
  - 确认无新的 parse error 或资源引用错误

- **文档联动**
  - 代码任务完成后，检查是否需要同步更新：
    - `plan.md`
    - `docs/dev_plan.md`
    - `docs/frontend_dev_plan.md`

- **原则应用记录**
  - 在任务说明、开发记录或提交说明中记录：
    - 本轮适用原则
    - 提交前检查维度是否已覆盖

原则上：

- 能自动验证的，不只靠记忆
- 不能自动验证的，通过“本轮适用原则 + 提交前固定维度”兜底
