# Cent Jours — 前端开发优先级计划

> **更新**: 2026-03-22 v7
> **当前分支**: `claude/review-project-plan-vgQTN`
> **目标**: 用最少轮次把 Godot 入口从“占位页 + 联调脚本”推进到“可展示、可讲解、可继续绑定数据”的主场景

---

## 开发原则

> 继承通用开发原则，详见 `docs/dev_plan.md`。

### 一、前端核心原则

> **先骨架，后润色**
>
> 主场景优先完成信息架构、布局层级、数据占位与交互流向，
> 再逐步补充视觉细节、动画和美术资源。
>
> 判断标准：如果一个界面还不能讲清“玩家现在看到什么、接下来能点什么”，就不应该先做细节打磨。

> **Godot 薄展示层**
>
> 前端只负责：
> - 场景结构与布局
> - 读取 `GameState` / `CentJoursEngine` 已暴露数据
> - 响应 `EventBus` 信号刷新 UI
> - 呈现占位图、图标、文案与交互状态
>
> 判断标准：一段逻辑在没有 UI 时仍然有意义，就不应该写在前端场景脚本里。

> **原型先行**
>
> `docs/ui_prototype.html` 是主场景的视觉蓝本。
> Godot 场景应优先复现其结构，而不是边做边重新设计。
>
> 当前主结构目标：
> - 顶栏 Top Bar
> - 地图区 Map Area
> - 右侧边栏 Sidebar
> - 底部决策托盘 Tray

> **组件复用优先**
>
> 已存在的 UI 组件必须优先落地使用：
> - `src/ui/components/rn_slider.gd`
> - `src/ui/components/decision_card.gd`
> - `src/ui/theme/cent_jours_theme.gd`
>
> 判断标准：如果一个已有组件能承担 70% 的需求，先接进去，不要新写一套。

> **开发态可视化**
>
> 在主场景未接完真实数据前，允许使用占位文本、假数据和开发面板，
> 但必须能一眼分辨哪些是“临时展示”、哪些已经接入引擎。

### 二、契约与验收原则

> **主场景验收优先于菜单完整性**
>
> 当前“主场景”指的是进入游戏后的核心 HUD / 战略界面骨架，
> 不要求先做完整标题菜单、设置菜单、继续游戏等外围流程。

> **每轮必须可见**
>
> 每一轮前端迭代完成后，必须满足：
> - 运行后能肉眼看到变化
> - 能说明增加了哪一块结构
> - 若有交互，能实际点击或触发

> **分辨率先服务开发**
>
> 当前开发分辨率以 `1280x720` 为主，
> 保证 Windows Godot 中可独立窗口调试，不以内嵌运行体验作为标准。

## 执行前检查

- 本轮是否直接服务于“呈现主场景”，而不是先扩展外围菜单或后端能力。
- 本轮是否定义了可见验收结果，例如新增一块 HUD、一个容器分区或一条可点交互。
- 本轮是否优先复用现有组件、主题和 `docs/ui_prototype.html`，而不是重新发明界面结构。
- 本轮是否保持 Godot 作为薄展示层，避免把核心规则、数值结算或状态源写进场景脚本。
- 本轮结束后，是否需要同步更新 `docs/dev_plan.md` 和 `plan.md` 里的进度说明。

## 目标定义

### 本阶段目标

在 Windows 版 Godot 中尽快看到一个**像游戏主界面的主场景**，要求：

- 一进入运行态，就能看到完整的四区布局
- 顶栏显示日期、阶段、Legitimacy、Rouge/Noir、核心资源占位
- 地图区有背景、标题和若干节点占位
- 右侧边栏有将领/事件/叙事信息占位面板
- 底部决策托盘显示至少 3 张 `DecisionCard`
- 主场景不再依赖 `engine_smoke_test.gd` 作为入口脚本

### 非目标

- 暂不追求完整主菜单动画
- 暂不追求真实地图美术
- 暂不追求完整设置页 / 存档页 / 新手引导
- 暂不追求最终品质的字体、图标、音乐和粒子效果
