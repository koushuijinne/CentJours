# Codex Autonomous Workflow

> **更新**: 2026-03-24
> **目的**: 把 `CLAUDE.md` 里的开发循环规则展开为可执行 SOP，支持“尽量零阻塞、不暂停、自动决策”的连续开发。
> **适用范围**: 本仓库的日常开发、接手、夜间推进、交接整理与发布前收口。
> **重要：1.用户已指定在必要时开启Subagent 2.每完成一轮任务后自动 `add / commit / push` 3.每完成一轮任务后自动总结压缩上下文并写入 `docs/codex_handoff.md`**

---

## 0. 最高优先级规则

- **本文件定义的零阻塞自动循环规则，优先级高于仓库内其他模板、便捷脚本、总结习惯和“先停下来汇报”的默认倾向。**
- 若与会话里的“先总结再说”“先写 handoff 再说”“先停在 final 再说”发生冲突，以本文件为准：**先完成一轮闭环，再立即进入下一轮。**
- **压缩上下文并把摘要写入 `docs/codex_handoff.md` 也是最高优先级规则的一部分。** 它不是可选整理动作，而是下一轮零阻塞续跑的前置条件。
- **Codex 不负责文案。** 涉及叙事、历史事件正文、营销文案、结局文本等写作任务时，Codex 只负责结构、触发、数据接入、验证与工程集成，不承担具体措辞撰写或润色。
- 不再依赖任何外部监督器；默认由当前会话内的 Codex 直接按本文件连续推进。
- 只有 `硬阻塞`、用户明确说“停”、或当前仓库中确实不存在更高价值且可执行的 `P0 / P1` 子任务时，才允许结束。

## 1. 默认执行模式

- 用户已授予仓库内文档与代码的完整读写权限；默认直接执行，不等待额外许可。
- 默认目标不是“做完一个回答”，而是“完成一轮高价值开发循环并自动决定下一轮”。
- 除非出现硬阻塞，不因“需要下一步指令”而停下。
- 默认在当前仓库内连续推进，不切换到 Linux / WSL Godot 无头验证路径。
- **总结、当前基线、handoff、commit/push 都不是停机条件**；它们只是继续下一轮前的中间动作。
- 只要仍存在无阻塞的 `P0 / P1` 子任务，默认必须立刻开下一轮，不能把“本轮已经说清楚了”当作结束理由。
- 只有 `硬阻塞`、用户明确说“停”、或仓库内当前确实没有更高价值且可执行的任务时，才允许结束本次回复。

## 2. 一轮标准开发循环

1. 读取最小必要上下文
   - 必读：`docs/development_principles.md`、`docs/dev_plan.md`、`docs/codex_handoff.md`
   - 按任务补读：相关 ADR、交接建议、目标模块代码
2. 建立当前真实基线
   - 看当前分支、工作区改动、关键文件行数、测试/验证现状
   - 明确哪些内容是“文档里写的”，哪些是“代码里真的存在的”
3. 自动选择最高价值任务
   - 先按本文件第 3 节优先级排序
   - 再按“能否形成闭环”“是否阻塞发布”“是否用户可感知”做二次排序
4. 直接实现
   - 小步推进，优先形成可验证闭环
   - 避免一轮混入不相干的大重构
5. 立即验证
   - Rust 改动跑 `cargo test`
   - UI / GDExt 改动走 Windows 原生验证
6. 同步文档
   - 更新 `docs/dev_plan.md`
   - 更新 `docs/codex_handoff.md`
   - 若默认流程变化，更新本文件
7. 提交并推送
   - 默认执行 `git add -A`
   - 按本轮独立范围写清晰 commit message
   - push 到当前工作分支
8. 压缩上下文并自动重排下一轮
   - 用 5-10 行写清本轮产出、验证结果、剩余最高优先级
   - 将压缩摘要写入 `docs/codex_handoff.md`
   - 扫描剩余缺口
   - 若当前主线已闭环，立刻切到下一条最高价值任务继续推进

## 3. 默认优先级重排规则

完成一轮后，按以下顺序自动选择下一轮任务：

1. 阻塞主循环、编译、测试或验证链路的问题
2. `docs/dev_plan.md` 中当前 `P0` 项
3. 能把一条垂直切片从“半成品”推进到“可演示/可提交”的任务
4. 用户可直接感知的高收益 UI / 文案 / 体验问题
5. 会放大后续返工成本的工程债
6. 文档、交接、工具链完善

如果两个任务价值接近，默认选择：

- 更接近 Steam 首发阻塞项的那个
- 更容易闭环验证的那个
- 更不依赖用户拍板的那个

## 4. 默认决策规则

### 4.1 何时直接做决定

以下情况默认不问用户，直接决策：

- 只是实现路径选择，不改变产品方向
- 只是文档收口、验证方式澄清、命名统一、文案去重
- 只是为了让当前任务形成完整闭环所需的最小补充
- 存在更小、更稳、更符合现有架构的实现方式

### 4.2 何时必须问用户

只有以下情况才停下来询问：

- 涉及删除大量现有内容或高风险回退
- 需要在多个产品方向之间取舍，后果明显不同
- 需要仓库外凭证、付费资源、第三方账户、Steam 后台人工操作
- 发现工作区现有改动与当前任务直接冲突，且无法安全避开

### 4.3 前端默认取舍

- 先修结构性问题，再修像素问题
- 先修阻碍阅读和操作的问题，再做纯视觉润色
- 若局部补丁连续两轮无效，默认升级为布局结构调整

### 4.4 架构默认取舍

- Rust 保持规则权威，GDScript 保持薄层
- 优先数据驱动，不把内容写死进 UI 脚本
- 小量重复优于过早抽象

## 5. 阻塞处理规则

### 5.1 软阻塞

定义：

- 解析失败
- 编译失败
- 测试失败
- Windows 验证命令失败但可定位

处理：

- 立即修复
- 修后重跑同一验证
- 不切换任务

### 5.2 中阻塞

定义：

- 当前主线依赖 Windows 真机结果，但还有其他高价值任务可继续
- 当前模块信息不足，但其他模块可以独立推进

处理：

- 在 `docs/codex_handoff.md` 记录阻塞与待验证步骤
- 立即切换到不依赖该阻塞的下一条任务
- 不因“主线暂时卡住”而停机

### 5.3 硬阻塞

定义：

- 需要用户做产品取舍
- 需要外部账户/支付/法律/平台后台操作
- 需要确认会不会覆盖用户现有工作

处理：

- 只问最小必要问题
- 同时给出推荐选项和理由
- 等待用户明确答复

## 6. 默认验证矩阵

### 6.1 Rust 规则层

- 默认命令：`cd cent-jours-core && cargo test`
- 任何影响规则、状态机、事件、叙事、存档的改动都要过 Rust 测试

### 6.2 Rust + GDExt API

- 先在当前环境跑 Rust 测试
- 需要 Godot 加载的改动，最终以 Windows 侧重编为准：

```bash
cd /d E:\projects\CentJours\cent-jours-core
cargo build --features godot-extension
```

### 6.3 GDScript / 场景 / UI

- 默认只走 Windows 原生验证
- 默认 Windows 无头命令：

```bash
E:\software\godot\Godot_v4.6.1-stable_win64_console.exe --headless --path E:\projects\CentJours --quit
```

- 视觉、布局、字号、滚动和动效以 Windows 真机截图 / 肉眼为最终准绳

### 6.4 明确禁止

- 不把 Linux / WSL 无头 Godot 当作默认验证路径
- 不用 Linux / WSL 无头结果替代 Windows UI / GDExt 验证结论

## 7. 文档同步规则

每完成一轮独立工作，默认同步：

- `docs/dev_plan.md`
  - 更新当前状态、优先级、技术债、验证方式
- `docs/codex_handoff.md`
  - 更新真实状态、已知问题、下一步建议、当前阻塞
  - 追加或覆盖“最近一轮压缩摘要”，确保下一轮可直接续跑
- `plan.md`
  - 仅在本轮确实改变产品里程碑状态时更新
- 本文件
  - 仅在自动开发规则本身变化时更新
- git 状态
  - 默认 `git add -A`、commit、push，让代码、文档与 handoff 一起落地

## 8. 工作区与安全规则

- 不回退用户已有未提交改动
- 不读取 `.env*`、密钥、证书、私密凭证文件
- 不使用破坏性 git 命令
- 默认在现有分支继续工作；若 handoff 中的分支名与实际不一致，以 `git branch --show-current` 为准

## 9. 何时算一轮完成

满足以下条件即可视为一轮完成：

- 当前目标有清晰产出，不再是半成品
- 已完成对应验证，或已明确记录无法验证的原因
- `docs/dev_plan.md` 与 `docs/codex_handoff.md` 已同步
- 已完成 `git add -A`、commit、push，或已明确记录无法 push 的原因
- 已产出压缩摘要，并已写入 `docs/codex_handoff.md`
- 已经选出下一轮最高价值任务

### 9.1 明确不算完成

以下情况 **都不算完成**，必须继续：

- 只是输出了当前状态总结、handoff 摘要或“下一步建议”
- 只是完成了 `git add -A`、commit、push
- 只是更新了文档，但还没有开始下一条无阻塞高价值任务
- 只是因为本条回复已经够长、够完整、够像收尾

如果提前做完，默认继续：

1. 扫描 `docs/dev_plan.md` 当前最高优先级
2. 找出最小可执行子项
3. 直接开始下一轮，不等待用户追加指令
