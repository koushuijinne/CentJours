# ADR-009: Tier 4 内容扩充与打磨路线图

## 状态: 已采纳

## 背景

M0–M4（Tier 0–3）全部完成：143 单元测试通过，核心循环（Dawn/Action/Dusk）、战斗解算、政治系统、命令偏差、将领网络、事件池分级、蒙特卡洛模拟、叙事引擎、GDExtension 绑定、Save/Load 均已实装。前端 F0–F4 达 95–100%，F5 视觉统一约 55%。

项目进入 **Tier 4 — 内容与打磨** 阶段。本 ADR 记录 Tier 4 四个优先级的修改方案、实施策略和验收标准。

---

## 修改方案总览

| 优先级 | 项目 | 规模 | 目标 | 决策理由 |
|--------|------|------|------|---------|
| **P1** | 历史事件扩充 31→100+ | L | 填满 100 天游戏跨度 | 31 条事件密度不足，Day 50–85 几乎空白；ADR-008 框架已就绪 |
| **P2** | F5 视觉统一 | M | 55%→80%+ | 托盘双滚动 + 中英混排是用户可感知的最直接问题 |
| **P3** | main_menu.gd 第五波解耦 | M | 543→<300 行 | 回合流/HUD 刷新混在编排器，改动风险高 |
| **P4** | Rust unwrap 清理 | S | 53 处→<10 处 | 长期债务，提升健壮性，可穿插完成 |

## P2: F5 视觉统一

### 现状分析

- 托盘（tray）双滚动问题：内外容器同时响应滚轮
- 中英混排：部分 label 英文、部分中文，字体/间距不统一
- 动效：仅有 ad-hoc flash（HUD 闪烁），无统一动效框架

### 实施方案

#### 2.1 托盘双滚动修复

- `tray_controller.gd` 中拦截内层 `ScrollContainer` 的滚轮事件
- 当内层可滚动时，阻止事件冒泡到外层
- 方案：重写 `_gui_input()` 或使用 `mouse_filter = MOUSE_FILTER_STOP`

#### 2.2 中英混排统一

- 建立 `src/ui/theme/` 下统一的 `FontConfig`
- 所有 Label 使用主题系统的 `default_font`，不单独设置
- 中文优先，英文专有名词保留原文但统一字体

#### 2.3 动效框架

- 创建 `src/ui/effects/transition_manager.gd`
- 统一入场/退场/高亮三类动效
- 用 `Tween` API 替换 ad-hoc Timer + modulate 闪烁
- 不引入第三方动效库（KISS 原则）

### 验收标准

- [ ] 托盘区域滚动不穿透
- [ ] 所有可见 Label 字体统一
- [ ] HUD 数值变化有统一 Tween 动效
- [ ] 1280×720 下无溢出/截断

## P4: Rust unwrap 清理

### 现状分析

- 全局约 102 处 `unwrap()`/`expect()`/`panic!()`
- 其中测试代码内的 unwrap 不需清理（测试中 panic 是正确行为）
- 生产代码中约 53 处需审查，集中在：
  - `events/pool.rs` 28 处（JSON 解析）
  - `lib.rs` 13 处（GDExtension Dictionary 访问）
  - `engine/state.rs` 11 处（状态反序列化）

### 实施方案

1. **测试代码中的 unwrap 保留**：不改
2. **JSON 解析路径**：替换为 `Result` 传播 + 顶层 `match`，解析失败返回 `Err` 而非 panic
3. **GDExtension 层**：`lib.rs` 中的 `unwrap_or()` 已是安全默认值，仅清理裸 `unwrap()`
4. **engine/state.rs**：序列化/反序列化路径改用 `?` 操作符
5. **逐文件提交**：每个文件单独一轮，改完立即 `cargo test`

### 验收标准

- [ ] 生产代码 unwrap 数量 < 10
- [ ] `cargo test` 143 测试全部通过
- [ ] 无新增 `panic!()` 路径

## 实施顺序与依赖关系

```
P1 (事件扩充)  ──────────────────────────────────►  可独立进行
P2 (视觉统一)  ──────────────────────────────────►  可独立进行
P3 (解耦)      ──── P2 完成后更稳妥 ───────────────►  软依赖 P2
P4 (unwrap)    ── 穿插在 P1/P2/P3 间隙 ──────────►  无依赖
```

- P1 和 P2 可并行
- P3 软依赖 P2（视觉统一后 controller 边界更清晰）
- P4 规模最小，穿插在其他任务间完成

---

## 后果

- **正面**：100+ 事件填满游戏跨度，视觉一致性提升第一印象，代码可维护性改善
- **风险**：P1 事件质量需严格过 ADR-008 Checklist，否则数量增加反而稀释体验
- **技术债**：P3 解耦可能暴露 controller 间的隐式依赖，需逐步验证
- **不做的事**：M5 美术替换、M6 音效、多语言、Steam 集成均不在本轮范围
