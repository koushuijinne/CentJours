# ADR-009: Tier 4 内容扩充与打磨路线图

## 状态: 已采纳

## 背景

M0–M4 已完成，核心循环已稳定运行。Dawn / Action / Dusk 主循环、战斗解算、政治系统、命令偏差、将领网络、事件池分级、蒙特卡洛模拟、叙事引擎、GDExtension 绑定以及 Save / Load 都已落地。

当前 Rust 规则层通过 `168/168` 测试。当前内容基线为 `15` 名角色、`41` 个地图节点、`58` 条历史事件。

前端 F0–F4 已联调完成，正式入口为 `src/ui/main_menu.tscn`。`historical_note` 已接入 UI 滚动日志，玩家在回合结算后能立即看到事件正文和史注。

项目现在进入 **Tier 4 - 内容扩充与打磨**。本 ADR 记录当前阶段的优先级、实施方向和验收标准。

---

## 决策总览

| 优先级 | 项目 | 规模 | 目标 | 决策理由 |
|--------|------|------|------|---------|
| **P1** | 历史事件扩充 | L | `58` → `100+` | 百日主线需要足够密度，当前内容量仍不足以撑起完整 100 天体验 |
| **P2** | 文案 QA 与史实收口 | L | 全量事件统一到 ADR-008 | 现有事件数量已足够进入系统性打磨，下一步要提升事实密度、句式一致性和视角覆盖 |
| **P3** | F5 视觉与布局收口 | M | 视觉统一、滚动稳定、文本层级清楚 | 托盘、地图面板、字体和动效仍需做发布级整理 |
| **P4** | 发布链路与工程收口 | M | Windows 导出、验收、清理技术债 | 当前已经可玩，下一步要把构建、导出、验证和交付流程固定下来 |

---

## P1: 历史事件扩充

### 现状

- 当前历史事件为 `58` 条，包含 major / normal / minor 三层。
- 事件分布已覆盖开局、巴黎阶段、备战期和终盘，但总量仍偏少。
- Day 50–85 已有内容基础，仍需要更密的政治、外交、联军和民间事件。

### 方向

1. 扩充到 `100+` 条事件。
2. 优先补齐中盘与终盘事件密度。
3. 继续补充 minor 事件，让时间线上有更稳定的日常噪声和民间视角。
4. 保持事件池为数据驱动结构，不改 JSON schema 的主形态。

### 内容切口

- 巴黎执政期的政治与外交事件
- 备战期的军队重编、要塞、后勤与联军动态
- 滑铁卢前后的关键战役节点
- 民众、官僚、流亡宫廷、联军参谋体系的旁支视角

### 验收标准

- [ ] 事件总数达到 `100+`
- [ ] Day 50–85 不再出现大段空白
- [ ] major / normal / minor 的分布与 ADR-008 体系一致
- [ ] `cargo test` 继续通过

---

## P2: 文案 QA 与史实收口

### 现状

- 当前内容已经开始按 ADR-008 进行直写化修订。
- 旧事件中仍残留少量文学化套话、事实锚点松散、视角单一的问题。
- 目前的任务重点是逐条收紧表达，而不是再做抽象化改写。

### 规则

1. 文案允许继续修改。
2. 所有历史事件正文和史注都要遵守 ADR-008 的直接、清楚、可考据原则。
3. 禁止使用 reframing 句式，包含“不是……而是……”“真正的问题是……”等表达。
4. 史注优先写事实、日期、因果，不写空泛评判。
5. 事件正文优先写具体场景，不写重复解释。

### 重点处理对象

- 外交事件的具体出使、回函和立场变化
- 议会和政治冲突的具体议题
- 联军视角事件的战役准备与行动路径
- 终盘事件的时间锚点和因果链

### 验收标准

- [ ] 全量历史事件符合 ADR-008 写作原则
- [ ] 历史事件正文没有明显 reframing 句式
- [ ] 史注统一为事实密度更高的短段落
- [ ] 关键事件的时间线和地点信息自洽

---

## P3: F5 视觉与布局收口

### 现状

- 主菜单控制器已经拆分，但发布级 polish 还没有完全收口。
- 托盘滚动、地图面板宽高、字体统一和文本层级仍是当前前端的主要收尾点。

### 方向

1. 保持现有 UI 结构，不再引入新的大改版。
2. 继续收紧滚动行为，避免双滚动和事件穿透。
3. 统一字体、字号和排版节奏。
4. 补足必要的入场、强调和状态反馈动效。

### 验收标准

- [ ] 托盘滚动稳定
- [ ] 地图与说明面板不挤压核心信息
- [ ] 可见文本风格统一
- [ ] Windows 1280×720 下布局可用

---

## P4: 发布链路与工程收口

### 现状

- 项目已经进入可连续开发阶段。
- 下一步要补的是导出、验收、资产清单和工程收口。

### 方向

1. 固化 Windows 导出链路。
2. 固化验证命令和发布前检查。
3. 把必要的技术债拆进独立小任务处理。
4. 保持 Rust / GDScript 的边界清楚。

### 验收标准

- [ ] Windows 导出链路可重复
- [ ] 关键验证步骤有明确记录
- [ ] 生产代码中的高风险 `unwrap` 持续减少
- [ ] 发布前检查项可执行

---

## 实施顺序

1. P1 历史事件扩充
2. P2 文案 QA 与史实收口
3. P3 F5 视觉与布局收口
4. P4 发布链路与工程收口

P1 和 P2 可以交错推进。P3 以现有 UI 结构为基础做收尾。P4 穿插进行，但不压过内容主线。

---

## 后果

- 正面：内容密度继续上升，百日主线更完整，历史叙事的可信度更高。
- 正面：文案风格会更统一，事件正文和史注的阅读成本会下降。
- 正面：前端和发布链路会逐步从开发态转向交付态。
- 风险：如果事件扩充只追数量，不做 QA，体验会变散。
- 风险：如果前端收口和发布链路被过早放大，内容主线会被打断。
