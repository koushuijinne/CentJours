# ADR-006: 主场景展示稳定性应分阶段优先于后续视觉打磨

## 状态: 已采纳

## 背景

Priority A-C 完成后，`main_menu.tscn` 已经从“占位页”推进到“可运行的主游戏界面”。  
但在 Windows Godot、`1280x720` 开发分辨率下，主场景出现了一组新的展示问题：

1. **顶栏垂直裁切**
   - `TopBar` 固定高度为 `74`
   - 但 `RNBlock` 使用“标题 + 36px 滑条”的竖排结构
   - 去掉上下 margin 后，可用高度不足，导致顶部内容被裁切

2. **决策托盘内容截断**
   - `DecisionTray` 固定高度仅 `188`
   - 单张 `DecisionCard` 最小高度为 `110`，且内部仍包含图标、标题、行动点和效果列表
   - 在当前托盘高度下，卡片底部和效果信息被截断

3. **Sidebar 空间挤压**
   - Sidebar 固定宽度为 `320`
   - 忠诚度面板显示前 8 名将领，长名字与状态标签会横向挤压
   - 上方面板高度增长后，`History & Narrative` 面板被压缩

4. **地图北部标签重叠**
   - 地图节点标签按线性归一化直接投放到画布
   - 当前没有碰撞规避、标签优先级或 hover 展示机制
   - 布鲁塞尔周边节点标签严重重叠

5. **信息层级和文案语言仍偏开发态**
   - `Current Situation` 中地点与 `Legitimacy` 同行显示，可读性一般
   - 顶栏、地图、侧栏混用英语 / 法语 / 中文，尚未形成稳定 UI 语言规范

这些问题的严重程度并不完全相同。  
其中，**顶栏裁切、托盘截断、Sidebar 挤压** 属于会直接破坏演示和使用体验的布局稳定性问题；  
而地图标签碰撞与文案统一更适合作为下一轮单独处理。

## 决策

采用 **三阶段修复顺序**，而不是一次性同时处理所有展示问题。

### 第一轮：布局稳定性优先

当前最高优先级只处理以下三类问题：

- 顶栏内容裁切
- 决策托盘内容截断
- Sidebar 忠诚度/叙事面板空间挤压

目标是先让主场景在 `1280x720` 下达到：

- 不裁切
- 不溢出
- 核心面板都可读

本轮不要求解决地图标签碰撞或语言统一。

### 第二轮：地图可读性

在布局稳定后，再处理地图北部标签重叠：

- 收紧默认显示标签的节点类型
- 引入重点节点常显、次级节点 hover/选中显示的策略
- 必要时对北部战区做局部去碰撞或偏移布局

### 第三轮：信息层级与文案统一

在前两轮完成后，再统一：

- `Current Situation` 的信息分层
- 顶栏 / 地图 / 侧栏的标题语言
- 开发态提示与正式 UI 文案的边界

## 第一轮范围定义

### 范围内

- `TopBar` 高度与内部控件排布
- `DecisionTray` 高度、滚动区可见面积、卡片内容密度
- Sidebar 宽度 / 面板高度分配 / 忠诚度面板显示策略
- 1280x720 下的整体可读性复核

### 范围外

- 地图节点 hover / 点击详情
- 地图标签碰撞规避
- 多语言文案统一
- 进场动画、hover polish、粒子/音效

## 替代方案（被否决）

| 方案 | 否决理由 |
|------|---------|
| 一次性同时修布局、地图、文案 | 作用面过大，难以归因，验证成本高 |
| 先做地图标签碰撞，再回头修布局 | 截图中的第一感知问题仍会存在，演示收益不高 |
| 先做动画和视觉 polish 掩盖问题 | 无法解决实际裁切和溢出，只会放大问题 |
| 继续维持当前布局，仅依赖更大分辨率 | 违背当前以 `1280x720` 作为开发基准的原则 |

## 后果

- ✅ 第一轮范围清晰，可在一次较小迭代内完成并验证
- ✅ 修复结果对演示质量提升直接、可见
- ✅ 之后地图与文案问题可以单独验收，不与布局问题互相干扰
- ⚠️ 第一轮完成后，地图北部标签重叠仍会存在
- ⚠️ 第一轮完成后，UI 语言仍可能存在中英法混用
- ⚠️ 需要同步更新前端相关计划文档和 `docs/plans/dev_plan.md`，保证文档与新优先级一致

## 复盘与经验沉淀（2026-03-22）

第一轮最终达标，但修复过程明显偏慢，主要暴露了 4 个问题：

1. **过早把问题判断成“像素值不对”**
   - 顶栏裁切、托盘贴边的根因其实是布局模型不对。
   - 前几轮修复仍停留在调 `custom_minimum_size`、margin 和字号，属于局部补丁，不是结构性解决方案。

2. **没有尽早检查组件内部约束**
   - `DecisionCard` 外部尺寸变化后，内部 UI 不会自动重建。
   - `LoyaltyScroll` 引入后，内容宽度没有显式绑定到侧栏可用宽度，导致名字列被压成 0。
   - 这些都说明仅看场景树外框还不够，必须同步检查组件内部最小尺寸与重建机制。

3. **无头测试被用在了不适合的问题上**
   - `godot --headless` 能快速发现脚本解析、路径和扩展加载问题。
   - 但它不能证明布局“视觉上已经对了”。
   - 对这类 UI 问题，Windows 目标分辨率截图应该更早进入验证闭环。

4. **没有设置“升级重构”的明确阈值**
   - 当同一块 UI 连续两轮仍依赖像素补丁时，应该立即升级为布局模型重构。
   - 这次直到第三轮才明确切换到“最小高度 + 弹性分配 + 局部滚动”，导致迭代轮次偏多。

### 后续执行规则

- 对顶栏裁切、底部贴边、面板挤压这类问题，先判断是“布局模型问题”还是“局部间距问题”。
- 先检查 `size_flags`、`custom_minimum_size`、滚动容器、组件内部重建逻辑，再决定是否调像素值。
- 若同一问题连续两轮未收口，默认升级为结构性重构，不继续叠加局部补丁。
- 对视觉问题，以 Windows 目标分辨率运行截图作为最终收口依据；无头测试只做脚本/加载兜底。
