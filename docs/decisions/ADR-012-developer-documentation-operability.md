# ADR-012: 开发者文档可接手性基线

## 状态

已采纳

## 日期

2026-03-27

## 背景

项目当前已经具备持续开发能力，也已经积累了计划、历史、规则和 ADR 文档。但这些文档主要服务连续开发中的接手 agent，不足以让第一次进入仓库的人类开发者快速完成以下任务：

- 理解项目是什么、当前做到哪里
- 在 Windows 环境把项目跑起来
- 找到正确的测试入口和质量门槛
- 理解 Rust / Godot / 存档 / 数据之间的边界
- 在发现 bug 时，用统一方式记录、追踪和回归

现状问题不是“没有文档”，而是“缺少面向人类开发者的统一入口和标准骨架”。信息目前分散在 `docs/plans`、`docs/history`、`docs/decisions` 和规则文档里，阅读成本偏高。

## 决策

### 1. 建立面向开发者的主入口文档

仓库根目录必须存在 `README.md`，并至少包含以下部分：

- 项目简介
- Windows-first 环境部署说明
- 运行方式
- 测试与质量门槛
- 当前 roadmap
- 术语表
- 文档导航

`docs/README.md` 保留为文档目录索引，但不替代根 `README.md`。

### 2. 补齐两份长期维护文档

新增并长期维护：

- `docs/architecture.md`
- `docs/interfaces.md`

其中：

- `architecture.md` 负责解释整体系统结构、运行时模块关系、核心数据流、存档与验证结构
- `interfaces.md` 负责解释 Rust / Godot 边界、`GameState` 同步契约、Save/Load 契约、测试与 CI 入口

这两份文档的目标是降低“读代码才能知道边界”的成本，而不是重复逐文件注释。

### 3. bug 文档制度正式化

`docs/bugs/` 由“问题截图堆放区”升级为正式 bug 记录区，至少包含：

- `bug_index.md`
- `bug_template.md`

后续 bug 记录统一采用结构化格式，至少写清：

- ID
- 严重级
- 复现步骤
- 预期结果
- 实际结果
- 影响范围
- 根因
- 修复状态
- 回归验证
- 附件

截图可以继续保留在 `docs/bugs/`，但必须能从索引文档追溯到具体问题条目。

### 4. 代码命名与注释策略

不采用“注释率至少 40%”作为硬性质量指标。理由如下：

- 注释覆盖率高不等于可维护性高
- 强行追求比例会制造大量重复代码字面含义的低价值注释
- 真正需要的是关键路径解释，而不是普遍性噪音

正式采用以下规则：

- 新增或修改的函数、方法、测试名，默认使用英文标识符
- 中文正文、中文注释、中文文档仍然保留，并继续作为项目主语言
- 对复杂状态机、存档兼容、GDExt 边界、非直观算法和 UI 交互链补中文解释
- 已存在的中文函数名不做一次性全仓重命名；后续在触达相关模块时渐进改成英文

### 5. 文档更新触发条件

发生以下变化时，不能只改代码，必须同步文档：

- 环境部署方式变化
- 测试与验证方式变化
- Rust / Godot 边界变化
- Save / Load 契约变化
- 核心玩法主循环变化
- bug 管理流程变化

### 6. 用 CI 强制执行文档同步

项目引入独立的 GitHub Actions 文档同步检查：

- 当 `src/`、`cent-jours-core/`、`tests/`、`tools/` 或 `.github/workflows/` 发生改动时
- 必须同时修改 `README.md` 或 `docs/`
- 否则 CI 直接失败

这条规则的目的不是增加形式负担，而是把“文档更新触发条件”从口头约定变成仓库级门禁。

## 后果

### 正面影响

- 人类开发者接手速度更快
- 测试入口、质量门槛和运行方式更容易被正确执行
- 关键系统边界不再只存在于实现细节里
- bug 记录、回归和规则优化可以形成统一闭环
- 代码和文档的漂移会更早暴露在 PR / push 阶段

### 代价

- 需要维护根 `README.md`、架构文档、接口文档和 bug 索引
- 每次核心边界变化时，多一层文档同步成本
- 部分旧函数名和旧注释风格不会一次性清空，需要接受渐进迁移
- CI 需要多执行一条轻量门禁脚本

## 实施顺序

1. 新建根 `README.md`
2. 新建 `docs/architecture.md`
3. 新建 `docs/interfaces.md`
4. 规范 `docs/bugs/`
5. 更新 `docs/plans/dev_plan.md`、`docs/history/agent_handoff.md`、`docs/README.md`
6. 在后续代码迭代里渐进推进“英文函数名 + 关键中文注释”
