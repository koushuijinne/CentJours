# ADR-011: 核心循环系统化程度与历史逻辑深度

## 状态: 已采纳

## 背景

`docs/advice/agent_chat_history.md` 对《Cent Jours》的核心玩法做了多轮横向比较。结论不是“像哪一款游戏”，而是当前项目已经形成了自己的方向：

- 时间尺度是 `100 天 / 100 回合` 的极端压缩战略循环
- 主体是“拿破仑本人”的政治-军事危机处理，而不是国家沙盒
- 当前最强的设计资产不是单一战斗系统，而是军事、政治、忠诚、后勤、历史叙事之间的数值与文本耦合

但同一份分析也指出了真实缺口：中后期张力、主动人事安排、历史逻辑层次、玩法重玩差异、Steam 首发所需的完成度，都还没有完全收口。

本 ADR 要把这些判断正式写成项目决策，避免后续迭代在“继续加系统”与“先把核心玩法做深”之间来回摇摆。

---

## 一、当前判断

### 1.1 系统化程度判断

当前项目的系统化程度是：

- `核心循环：高`
- `跨系统耦合：高`
- `可演示完整度：中`
- `可上线完整度：未达标`

分项判断如下：

| 维度 | 当前判断 | 依据 |
|------|----------|------|
| 核心循环压缩感 | 高 | 100 回合内完成完整政治-军事危机，行动点稀缺且每天都有机会成本 |
| 军事-政治-忠诚耦合 | 高 | 胜败、合法性、派系、命令偏差、叛逃和后勤都能相互传导 |
| 后勤系统化 | 中高 | 已有节点容量、补给角色、补给窗口、阶段目标、三日节奏、区域任务 |
| 角色系统化 | 中 | 已有忠诚、性格、偏差、关系，但玩家主动“人事安排”的决策维度还不够厚 |
| 历史叙事系统化 | 中高 | `historical_note`、事件 tier、行动结算日志和叙事池已贯通，但文本视角与史实层次仍需统一 |
| 中期玩法张力 | 中低 | Day 20-80 仍有重复行动风险，重玩差异还不够强 |
| 产品化完整度 | 中低 | 主菜单、测试、存读档、日志链已有基础，但教程、结局、设置、资产、Steam 面缺口明显 |

### 1.2 历史逻辑深度判断

当前历史逻辑的体现深度不是平均的，而是“局部很深、整体还不均匀”。

已经体现较深的部分：

- 命令摩擦：忠诚度、性格和命令偏差已经把“拿破仑下令 != 现实必然执行”做成机制
- 时间压力：反法同盟增长、关键事件窗口和 100 天倒计时都在持续施压
- 后勤约束：前线消耗点、区域整补点、战略枢纽、补给窗口已经让“为什么不能一直北上”变成可见规则
- 政治张力：四派系与 Rouge / Noir 已经把“回到巴黎后不是只剩打仗”做成可运算结构

仍显偏浅的部分：

- 人事与大本营逻辑：玩家对将领、顾问、内阁、议会人物的主动调度空间还不够
- 社会代价链：征兵、军费、补给、民心之间已有数值反馈，但缺少更强的社会后果层
- 历史层次感：外部视角、档案视角、拿破仑主观视角的边界刚开始成型，尚未全量统一
- 战役具体性：天气、兵员质量、新兵与老兵差异、战役经验等历史关键因素还没有形成稳定规则层

结论：

- 当前项目已经不是“历史题材皮肤 + 抽象策略数值”
- 但还没有达到“每个关键玩法都能自然解释其历史原因”的程度

---

## 二、决策

### 2.1 产品定位锁定

本项目的正式定位锁定为：

> **100 天高压政治-军事危机模拟器**

这意味着：

- 不走 P 社式长线沙盒
- 不走全面战争式战术微操
- 不把重点转向大地图扩张或完整外交树
- 继续围绕“短周期、高压、可追责、可复盘”的核心循环做深

### 2.2 Steam 可上线前的完成标准

项目的近期终局目标正式定义为：

> **核心玩法优化完成，并达到 Steam 可上线级别**

这里的“可上线级别”至少包含 5 个条件：

1. 核心循环清晰
   - 新玩家在前 10 天能理解“为什么要补给、为什么要稳政治、为什么将领会出偏差”
2. 至少存在 3 条可行玩法路径
   - 军事优先、政治优先、均衡运营都能成立，不能只剩单一最优解
3. 历史逻辑可感知
   - 玩家能从系统与文本里读出“百日王朝为什么会走到这一步”
4. Windows 验证链稳定
   - Rust、`GdUnit4`、smoke、Windows CI 能稳定兜住主链路回归
5. 首发表面完整
   - 新局、存读档、教程、结局、设置、关键 UI、基础资产和 Steam 页面材料可交付

### 2.3 叙事视角决策

叙事视角采用“混合分层”，不是单一视角覆盖全部文本。

具体规则：

- 拿破仑第一人称：
  - 用于教学、行动建议、阶段复盘、结局前短独白等玩家主观层
- 第三人称 / 档案体：
  - 用于 `historical_note`、联军情报、议会动向、外交动态、外部观察
- 同一条信息既要代入又要考据时，拆成两层表达，不混写

### 2.4 大本营互动场景的优先级

用户提出的“大本营第一/第三人称互动场景”方向有价值，但当前阶段定性为：

- `低优先级`
- `不进入 Steam 首发 P0`

原因：

- 它会引入新的交互层、空间层、角色关系层和内容制作成本
- 当前项目更缺的是核心循环厚度、历史逻辑统一、教程和验证链稳定，不是新的场景载体

正式决策：

- 在 Steam 首发前，只允许做轻量化的大本营表达，例如：
  - 文本化会面
  - 人事任命界面
  - 角色来信 / 情报回报
  - 简化的恢复 / 关系 / 情报选择
- 不以“可自由走动的大本营场景”作为首发前主线

---

## 三、后续优化顺序

在“核心玩法优化完成并达到 Steam 可上线级别”之前，优化顺序固定为：

1. 先稳验证链
   - Windows CI、`GdUnit4`、smoke、关键 bug 回归
2. 再补核心循环厚度
   - 中期张力、玩法路径差异、失败归因、教学链
3. 再统一历史逻辑表达
   - 视角分层、史实锚点、叙事密度、事件 QA
4. 再做首发表面完整度
   - 设置、结局、关键 UI、资产、Steam 页面
5. 最后再考虑低优先级扩展层
   - 大本营空间化互动、更多视角表现、非核心沉浸层

---

## 四、后果

正面影响：

- 项目不会因为“还能加很多系统”而偏离主线
- Steam 首发前的判断标准更清晰
- 文案、玩法、测试、产品面能围绕同一终局目标收口
- 拿破仑第一人称和历史客观层不会再互相抢位置

代价：

- 一些看起来很有吸引力的新层，如大本营场景化互动，会被明确推迟
- 文案与系统迭代要更受约束，不能随意混视角
- 后续每一轮自动工作流都应该优先服务“核心玩法优化 + Steam 可上线”这一终局，而不是局部漂亮但不影响上线的新增想法
