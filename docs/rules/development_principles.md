# Cent Jours — 开发原则

> **更新**: 2026-03-28
> **执行检查清单**: [execution_checklist.md](execution_checklist.md)

本文件沉淀长期稳定的项目原则。不承载短期任务、实现步骤或优先级排序。

---

## 1. 核心产品原则

### 1.1 核心循环优先

玩家能在 30 秒到 2 分钟内完成一轮"看情报 → 做选择 → 承担后果"。新功能优先服务 Dawn / Action / Dusk 循环，不为外围系统拖慢主循环验证。

### 1.2 有趣选择与可见后果

每个选择同时包含收益、代价和上下文信息。玩家必须能理解"为什么结果是这样"。政策、战斗、忠诚度、历史事件必须能被解释。

### 1.3 玩家心理优先于绝对写实

玩家失败后必须能归因。不为了写实牺牲可理解性和公平感。命令偏差、随机性、冷却、事件触发都要可读。

### 1.4 紧迫感优先于复杂度堆砌

"100 天、每天都重要"是核心差异化。新系统必须强化有限时间决策压力，不引入削弱节奏的低价值复杂度。

### 1.5 历史感服务于游戏体验

叙事、日记、历史节点必须和机制相互支撑。历史元素优先做"可感知价值"，不是百科全书。

## 2. 游戏设计原则

### 2.1 目标—障碍—工具

每个阶段明确告诉玩家：现在该做什么、为什么难、手里有什么手段。

### 2.2 可读性与可预判性

重要信息一眼可见，下一步风险能提前预判。顶栏、边栏、地图、托盘形成稳定信息层级。

### 2.3 教学嵌入游玩

玩家通过前几回合自然学会系统，不需要长篇阅读才能开始玩。首批可玩路径必须足够短。

### 2.4 反馈至上

每次操作后立刻收到视觉、文字、音效或状态反馈。选中态、数值闪烁、结算结果属于核心体验，不是装饰。

### 2.5 失败必须可归因

"输了但知道为什么"驱动再来一局。政治崩溃、战败、叛逃、结局都要有清晰解释。

## 3. 前端与体验原则

### 3.1 原型先行，先骨架后润色

先验证结构和节奏，再投入视觉资源。当前问题是"看不懂/不能用"时先修结构，不先磨像素。

### 3.2 Godot 薄展示层

前端负责展示与交互。无 UI 时仍有意义的逻辑不应写在 GDScript 场景脚本里。规则在 Rust，状态通过 `TurnManager → GameState → UI` 流动。

### 3.3 组件复用优先

同类交互和视觉优先复用现有组件（`rn_slider.gd`、`decision_card.gd`、主题系统），不重复造轮子。

### 3.4 信息层级优先

顶栏负责全局状态，边栏负责解释与摘要，托盘负责选择。最重要的信息位于最显著位置。

### 3.5 结构问题优先于像素补丁

UI 裁切、溢出和挤压先检查布局模型（`size_flags`、`custom_minimum_size`、滚动容器），再调像素值。连续两轮局部补丁无效应升级为结构性重构。

## 4. 工程与架构原则

### 4.1 TDD

Rust 规则层默认 Red → Green → Refactor。`cargo test` 通过是提交业务逻辑的默认门槛。

### 4.2 单一状态源

Rust 引擎是真实规则状态。`TurnManager` 负责同步，`GameState` 是 UI 只读缓存。Godot 侧不自行推导合法性、冷却或战斗结果。

### 4.3 GDScript 薄层

GDScript 负责 UI、信号、桥接和展示元数据。核心规则、事件效果、战斗计算、冷却结算留在 Rust。

### 4.4 DRY

战斗、政治、事件计算只在一处权威实现。GDExtension 返回键名和状态结构避免多处散写。

### 4.5 常量集中管理

游戏规则常量在 Rust 集中定义，通过接口暴露给 Godot。冷却时间、忠诚度阈值、阶段键名不允许多脚本散写。

### 4.6 YAGNI

不为"也许以后会用到"提前抽象。三行可读重复好过一个过早的万能系统。

### 4.7 KISS

数据流单向：引擎 → TurnManager / GameState → UI。场景脚本优先清晰直白。

### 4.8 数据驱动设计

`characters.json`、`historical.json`、`map_nodes.json`、`design_tokens.json` 是核心数据源。策划和平衡修改尽量不改引擎代码。

### 4.9 边界契约注释

GDExtension 暴露给 GDScript 的结构应有轻量契约注释。返回键名、类型和关键语义要在调用侧或桥接层写清。

### 4.10 ADR 记录重大决策

跨迭代、影响后续实现的重要选择通过 ADR 固化。见 [ADR 汇总表](../decisions/README.md)。

## 5. 迭代与验证原则

### 5.1 原型先于资产投入

新机制先用占位 UI 和假数据跑通。主循环未稳定前不优先投入高成本美术。

### 5.2 垂直切片优先

优先做"可跑的一整条链路"（输入 → 引擎结算 → UI 刷新 → 叙事反馈），不先各做半截。

### 5.3 小步迭代，尽快试玩

单轮改动足够小，能明确归因好坏。每轮能快速验证：变得更可玩了，还是只是更复杂了。

### 5.4 测试与试玩并重

自动化测试证明行为正确，试玩验证体验成立。两者不可互相替代。

### 5.5 每轮必须有可见成果

每轮至少产出：场景变化、交互闭环、测试新增或文档状态更新之一。

### 5.6 视觉问题以目标平台真机收口

Windows Godot 1280x720 运行截图是前端布局问题的默认收口标准。`--headless` 只用于脚本解析和资源装载兜底。

### 5.7 交接文档随轮次同步

完成一轮后同步更新 `agent_handoff.md`。新会话启动方式变化时同步 `agent_session_prompts.md`。

---

## 6. 禁止事项

- 不把核心规则写进 GDScript 场景脚本
- 不让前端自行维护引擎已有状态
- 不为历史写实牺牲公平感与可读性
- 不在核心循环未验证前堆砌外围复杂度
- 不让文档长期落后于真实实现
- 不放任 `agent_handoff.md` 与真实状态漂移
