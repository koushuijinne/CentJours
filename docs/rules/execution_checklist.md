## 1. 原则分级

原则分三类，决定检查频率：

### A 类：硬约束（每次必查）

- GDScript 薄层
- 单一状态源
- 数据驱动设计
- 文档同步

### B 类：默认适用（大多数任务查）

- 核心循环优先
- 有趣选择与可见后果
- 可读性与可预判性
- 先骨架，后润色

### C 类：按任务触发

- 反馈至上
- 教学嵌入游玩
- 垂直切片优先
- 历史感服务于游戏体验

## 2. 开工前：动态选择适用原则

每个任务开工前，根据改动内容动态选择本轮适用原则：

- 至少选 1 条，通常 2-5 条
- 跨层/跨系统任务可超过 5 条
- 不允许每轮机械固定选 3 条

### 参考选择模板

| 任务类型 | 建议原则 |
|----------|----------|
| Rust 规则/数值/事件 | 核心循环优先、有趣选择、数据驱动、TDD |
| Godot UI/交互 | 可读性、反馈至上、信息层级、薄展示层 |
| 新手体验/节奏 | 教学嵌入游玩、玩家心理优先、核心循环优先 |
| 跨层接口/状态同步 | 单一状态源、边界契约注释、薄层、DRY |

开始实现前，明确写出"本轮适用原则"。

## 3. 提交前：五维度检查

围绕 5 个固定维度动态生成检查问题：

| 维度 | 核心问题 |
|------|----------|
| **产品/玩法价值** | 是否强化了核心循环、选择质量或节奏张力 |
| **玩家体验/可读性** | 玩家能否看懂发生了什么、为什么、下一步该做什么 |
| **架构边界** | 是否违反薄层、单一状态源、数据驱动或分层职责 |
| **验证闭环** | 是否有可测试、可试玩、可观察的验证方式 |
| **文档同步** | 是否需要同步 dev_plan / agent_handoff / product_plan |

简单任务每维度 1 问，复杂任务可展开多问。

## 4. 自动化验证

能自动化的不靠记忆：

| 改动类型 | 自动化检查 |
|----------|------------|

| GDScript/场景 | Godot 项目加载 + GdUnit4 |
| 文档联动 | 检查是否需同步 dev_plan / agent_handoff / product_plan |
