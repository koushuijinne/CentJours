# Codex Overnight Plan

> **更新**: 2026-03-23
> **适用窗口**: 00:24 - 08:00 夜间连续开发
> **执行分支**: `codex/overnight-project-advance`
> **执行方式**: 当前 Codex 会话持续推进；若会话、终端或机器中断，本计划自动失效

---

## 1. 计划目标

本轮夜间开发不再只盯主菜单局部打磨，而是从**整个项目视角**推进到下一个更有价值的状态：

- 保住当前 Windows 可运行的最小可玩垂直切片
- 启动 `Tier 2` 的真实跨层接入，优先打通行军系统的最小闭环
- 在 `Tier 2` 骨架成立后，前推可提前独立完成的 `Tier 3` 子任务
- 持续推进主菜单第三波解耦，不让高收益工程债在夜间被遗忘
- 将结果沉淀到代码、验证结果和交接文档中，确保早上可以人工接手

## 2. 夜间开发总原则

- `项目级推进优先`：以整个项目的下一阶段价值排序，不只做单点 UI polish
- `先接骨架，再上深度`：先补 `Tier 2` 空间层骨架，再推进 `Tier 3`
- `主菜单解耦常驻`：主菜单第三波收尾是夜间循环中的常驻任务，不因 Tier 2 / Tier 3 主线而被完全搁置
- `不破坏主循环`：不允许为了新功能打断现有 Dawn / Action / Dusk 可玩链路
- `Windows 验证优先`：无头测试只跑 Windows，不跑 Linux / WSL
- `活文档`：每完成一轮独立推进后，同步更新 `docs/codex_handoff.md`
- `自动重排优先级`：每完成一轮后，默认重新扫描代码与文档，自动决定下一轮最高价值任务
- `禁止自然收口停机`：夜间模式下，"这一轮已经比较完整" 不是合法停止理由；完成一轮后必须直接进入下一轮，除非命中白名单停机条件

## 3. 夜间主线任务

### 第一阶段：冻结基线与确认现状

- 创建独立分支，避免污染白天工作分支
- 确认当前计划文档、handoff 文档与代码状态一致
- 确认当前主循环、主菜单和 Windows 验证链路仍可继续推进

### 第二阶段：Tier 2 最小闭环

目标：让“位置”从纯展示变成真实状态。

优先顺序：

1. Rust `GameEngine` 接入 `PlayerAction::March`
2. GDExt 暴露 `process_day_march()`、`get_adjacent_nodes()`、`napoleon_location`
3. `TurnManager` 接入 `march`
4. 主菜单与地图交互接入最小行军流程

最小完成标准：

- 玩家能在地图上选择相邻节点并提交行军
- 行军后 `napoleon_location`、疲劳、士气得到真实同步
- 主循环继续正常推进

### 第三阶段：Tier 3 前推

原则：只推进**不要求再次推翻 Tier 2 结构**的子项。

优先考虑：

- `Tier 3.3` 联军动态化
- `Tier 3.5` 事件效果补完
- `Tier 3.4` 存档 / 读档 UI（若前两项顺利）

暂不优先：

- `Tier 3.1` 命令偏差接入战斗
- `Tier 3.2` 叛逃 / 倒戈触发

原因：

- 这两项依赖位置上下文与行军链路更稳定后再做更稳
- 夜间无人值守时，不适合先做高解释成本的复杂行为联动

### 第四阶段：主菜单解耦与高收益工程债

该阶段不是一次性任务，而是夜间循环中的**常驻回退主线**。

优先考虑：

- `src/ui/main_menu.gd` 继续压纯，向纯编排器靠拢
- 主菜单子模块接口命名、状态桥接、职责边界收口
- 不改变产品方向前提下的高收益 UI / 工程债修复

触发时机：

- 当前主线完成一个可验证增量后，需要立刻切到下一轮最高价值任务
- 或主线推进暂时被更高层阻塞
- 或当前最有价值的下一步不是继续加系统，而是先降低维护成本

### 第五阶段：验证与收口

- Rust 改动执行 `cargo test`
- GDScript / 场景改动执行 Windows 无头测试
- 修复阻塞级 parse / compile / load 错误
- 同步更新：
  - `plan.md`
  - `docs/dev_plan.md`
  - `docs/frontend_dev_plan.md`
  - `docs/codex_handoff.md`

## 4. 明确不做的事情

- 不主动重写核心玩法方向
- 不在 GDScript 层复制 Rust 规则逻辑
- 不把 Linux / WSL 无头测试纳入默认验证
- 不做需要强审美拍板的大范围视觉改版
- 不为了追求“看起来做了很多”而无边界扩散任务

## 5. 夜间循环机制

夜间开发不是只跑一轮线性计划，而是按下面的循环持续执行：

```text
while 会话存活 and 无硬阻塞:
  1. 读取 plan / dev_plan / frontend_dev_plan / codex_handoff
  2. 扫描代码状态、验证状态与文档漂移
  3. 选择当前全项目最高价值且可立即推进的一轮任务
  4. 只实现一个最小可闭环增量
  5. 跑对应验证（cargo test / Windows 无头 / 真机截图待后补）
  6. 更新文档与 handoff
  7. 自动扫描下一轮优先级
end
```

这是一种“**不停机循环**”，不是“无边界乱做”：

- 每轮都必须有清晰开始与清晰收口
- 每轮结束都必须重新判断下一步，而不是机械按旧清单往下抄
- 只有在通过验证并同步文档后，才进入下一轮
- 禁止因为“当前轮达到自然收口点”就直接结束整个夜间会话
- 夜间模式下不应主动发送总结式停机消息，而应只做短进度更新并继续下一轮

## 6. 自动扫描下一轮优先级

每完成一轮后，默认执行以下扫描与重排：

1. 扫描代码层阻塞：
   - parse / compile / load 错误
   - 当前主循环是否被打断
   - 新增功能是否造成回归
2. 扫描计划文档漂移：
   - `plan.md`
   - `docs/dev_plan.md`
   - `docs/frontend_dev_plan.md`
   - `docs/codex_handoff.md`
3. 重新排序下一轮优先级，默认顺序为：
   - A. 阻塞主循环或验证链路的错误
   - B. `Tier 2` 骨架缺口
   - C. 可独立前推的 `Tier 3` 子项
   - D. 主菜单解耦与高收益工程债
   - E. 文档、工具链、交接与验证脚本完善

若扫描后发现原本的主线不再是最高价值任务，应立即切换，不强行执行过期计划。

## 7. 阻塞分级与处理策略

夜间循环默认**不会因为单个任务被卡住就整夜停机**。  
阻塞应先记录，再判断是否影响整个夜间循环。

### 7.1 阻塞分级

- `软阻塞`
  - parse error
  - compile error
  - 单测失败
  - 局部接口不匹配
- `中阻塞`
  - 当前主线暂时推不动，但仓库里仍有其他独立高价值任务
- `硬阻塞`
  - 需要人工拍板的产品分叉、视觉取舍或架构决策
  - 若继续开发会明显提高误判风险
- `环境阻塞`
  - 会话中断
  - 机器休眠 / 终端关闭
  - 验证环境失效
  - 路径、权限或平台能力缺失

### 7.2 默认处理规则

遇到 `软阻塞`：

1. 先做最小修复
2. 重新验证
3. 若仍失败，再降级为中阻塞处理

遇到 `中阻塞`：

1. 记录当前卡点
2. 更新 `docs/codex_handoff.md`
3. 立即切到不依赖该阻塞的下一个高价值任务

遇到 `硬阻塞` 或 `环境阻塞`：

1. 不默认整夜停机
2. 先记录：
   - 阻塞类型
   - 当前任务
   - 已尝试动作
   - 恢复条件
3. 重新扫描下一轮优先级
4. 若存在不依赖该阻塞的独立高价值任务，则继续下一轮
5. 只有在以下两种情况之一才停止：
   - 阻塞是全局性的，导致后续任务普遍不可执行
   - 继续开发将进入高风险盲改状态

### 7.3 阻塞后的循环动作

```text
遇到阻塞
  -> 记录到 handoff
  -> 判断是否为全局阻塞
  -> 若不是，全量重排下一轮优先级
  -> 继续执行下一个独立高价值任务
  -> 仅在全局阻塞或高风险盲改时收口
```

## 8. 夜间执行顺序

```text
分支与基线确认
  -> Tier 2 行军骨架打通
  -> 主菜单 / 地图最小行军交互接入
  -> 若主线稳定，继续吞 Tier 3 可独立子项
  -> 验证
  -> 文档与 handoff 收口
  -> 自动扫描下一轮优先级
  -> 不允许自然停机；必须切到下一轮最高价值任务
  -> 若当前主线不再是最高价值任务，切回主菜单解耦 / 工程债
  -> 继续下一轮
```

## 9. 验证规则

### Rust

- 默认执行 `cargo test`
- 若涉及状态序列化、引擎动作或事件效果，优先补最小必要测试

### Godot / GDScript

- 默认执行 Windows 无头测试：

```bash
E:\software\godot\Godot_v4.6.1-stable_win64_console.exe --headless --path E:\projects\CentJours --quit
```

- 只把无头测试用于脚本解析、资源装载、扩展初始化兜底
- 视觉与交互问题最终仍以 Windows 真机截图收口

## 10. 停止条件

夜间自动开发在以下条件之一触发时停止：

- 用户明确要求停止
- 遇到**全局性**阻塞，已无可安全切换的后续任务
- 会话、终端、机器或验证环境中断，且无法继续推进独立任务
- 继续开发将进入高风险盲改状态，且不存在可安全切换的下一轮任务

## 10.1 白名单停机条件

夜间模式下只允许因为以下原因停止：

- 用户主动打断或明确要求停止
- 会话 / 终端 / 机器 / 网络中断
- 出现全局阻塞，且已无独立高价值任务可切换
- 后续所有高价值任务都依赖人工拍板
- 继续开发会进入高风险盲改

除上述情况外，默认动作都是：

```text
完成当前轮
  -> 验证
  -> 更新 handoff
  -> 重排下一轮优先级
  -> 继续
```
